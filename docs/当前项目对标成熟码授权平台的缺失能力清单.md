## 1. 文档目的

本文件用于回答一个更偏产品规划的问题：

> **当前项目虽然已经具备码中心、订阅、资格和基础支付能力，但如果对标成熟的“优惠码平台 / 订阅码平台 / License 授权平台”，现在还缺哪些关键能力？**

本文件不是重新设计系统，而是明确：

- 当前项目**已经具备什么**
- 当前项目**真正缺什么**
- 这些缺口为什么重要
- 哪些缺口适合你现在的业务现实
- 后续应该按什么优先级推进

## 2. 评估基准

本次对标不是凭印象，而是参考了成熟平台的公开官方文档能力模型。

### 2.1 对标对象

- **Stripe**：偏支付后折扣、优惠券、promotion code、订阅折扣规则
- **Shopify**：偏电商优惠码、客户分群、组合规则、销售渠道与报表
- **Voucherify**：偏营销券、活动模板、发放分发、元数据与验证规则
- **Keygen**：偏软件授权、机器激活、Entitlement、Webhook、离线授权

### 2.2 为什么选择这四类作为基准

因为你的目标不是单纯做一个“输入兑换码的页面”，而是逐渐走向：

- 用户中心
- 支付/订阅中心
- 授权中心
- 码中心
- 产品接入中心

这正好横跨：

- 优惠与兑换
- 订阅发权
- 外部购买映射
- 授权生命周期管理

## 4. 总体判断

一句话总结当前状态：

## **当前项目已经有“码管理后台”，但还没有形成完整的“码运营与授权闭环”。**

目前更像是：

- 能建码
- 能查码
- 能导入导出
- 能做基础发权

但距离成熟平台通常还差：

- 发放层
- 申领审核层
- 撤销 / 补发 / 回滚
- 产品接入通知
- 模板 / 活动 / 批次正式化
- 规则引擎
- 运营报表
- 设备授权

因此，接下来不是继续堆更多相似页面，而是补中间那条真正让业务跑顺的链路。

## 5. 当前最关键的缺失能力（按优先级分层）

---

## 5.1 P0：必须优先补齐的能力

这些能力会直接决定项目能不能真正适配你现在的淘宝 / 闲鱼 / 外部发货 / 人工审核业务。

### 5.1.1 发放层 / 发布层（Publication / Delivery）

#### 当前缺口

当前系统已经有“码”，但还缺“把码正式发给谁”的结构化能力。

当前字段更多只是：

- 批次标签
- 渠道标签
- 外部订单号标签

但还没有一个真正的“发码记录 / 投递记录 / 领取记录”。

#### 成熟平台通常会有的能力

- 码绑定用户
- 码绑定订单
- 码绑定渠道
- 已发布 / 已发送 / 已领取 / 已使用 / 已撤回状态
- 发码记录追踪
- 重发与补发机制

#### 为什么你现在最需要它

因为你的真实业务场景不是纯在线支付，而是：

- 用户在淘宝 / 闲鱼购买
- 系统或人工发一个码
- 用户回站内兑换
- 系统发权

如果没有发放层，你很难稳定处理：

- 这个码发给谁了
- 同一订单是否重复发码
- 用户说没收到码时如何补发
- 发错了如何撤回

#### 建议补充的数据对象

建议至少增加：

- `code_publications`
- `code_deliveries`
- `code_claim_links`（或同等语义结构）

建议字段至少包括：

- `code_type`
- `code_id`
- `publication_status`
- `target_user_id`
- `target_contact`
- `source_platform`
- `external_order_no`
- `delivered_by`
- `delivered_at`
- `delivery_channel`
- `revoked_at`
- `revoke_reason`

---

### 5.1.2 订单申领 / 审核发权流程

#### 当前缺口

当前项目已经有兑换与订阅发权的基础，但还缺：

- 用户提交外部订单凭证
- 后台审核
- 审核通过后发码 / 发订阅 / 发资格

#### 为什么这是你业务场景里的关键能力

你当前并不稳定依赖在线支付回调，而更接近：

- 外部平台下单
- 站内申领
- 后台核对
- 发权完成

这意味着“订单申领”不是可选功能，而是你绕过支付回调限制的现实解法。

#### 建议补充的数据对象

建议增加：

- `order_claims`
- `order_claim_attachments`
- `order_claim_reviews`

建议状态包括：

- 待提交
- 待审核
- 审核通过
- 审核驳回
- 已发权
- 已撤销

建议字段包括：

- `source_platform`
- `external_order_no`
- `proof_images`
- `buyer_contact`
- `claimed_product`
- `claim_status`
- `reviewer_id`
- `review_notes`
- `granted_code_id`
- `granted_subscription_id`

---

### 5.1.3 撤销 / 补发 / 回滚

#### 当前缺口

当前项目已经支持：

- 启用 / 禁用
- 删除
- 批量状态修改

但这些还不是成熟业务意义上的：

- 已发放后的撤销
- 已授权后的回收
- 发错后的补发
- 退款后的回滚

#### 为什么这是必要能力

现实中一定会遇到：

- 发错套餐
- 发错用户
- 用户退款
- 订单取消
- 码已发出但需要重发

如果只有“disabled”而没有“业务语义上的撤销与回滚”，后续会越来越乱。

#### 建议补充的能力

- 码撤销
- 订阅回收
- 补发新码并作废旧码
- 审核驳回后释放库存
- 操作级回滚记录

---

### 5.1.4 产品接入回调 / 事件通知（Webhook / Event）

#### 当前缺口

当前系统内部可以变更状态，但产品侧不一定能自动知道。

#### 为什么必须补

如果未来要做到：

- 用户兑换后，产品自动解锁
- 用户订阅到期后，产品自动降级
- 用户资格被撤销后，产品自动失效

那么后台就必须有标准事件输出能力。

#### 建议至少支持的事件

- `code.redeemed`
- `registration_code.used`
- `subscription.granted`
- `subscription.revoked`
- `claim.approved`
- `claim.rejected`
- `entitlement.granted`
- `entitlement.revoked`

#### 最小落地建议

先做：

- 站内事件表
- Webhook 重试机制
- 基础签名校验
- 简单事件查询接口

---

### 5.1.5 模板 / 活动 / 批次正式化

#### 当前缺口

当前已经有：

- `batch_no`
- `campaign_name`

但它们目前更多是“码上的字段”，还不是独立的一层运营对象。

#### 成熟平台通常会怎么做

一般会拆成：

- 模板（Template）
- 活动（Campaign）
- 批次（Batch）
- 实例码（Code）

#### 为什么你也需要这一层

因为未来你会反复做：

- 淘宝活动
- 闲鱼补偿发码
- 老用户续费活动
- 邀请奖励活动
- 节日促销活动

如果不把模板 / 活动 / 批次独立出来，后面很多统计和复用能力都做不稳。

#### 建议的数据层结构

- `code_templates`
- `code_campaigns`
- `code_batches`
- `redemptions / registration_codes / subscription_codes`

---

## 5.2 P1：很重要，但可以在 P0 之后推进

这些能力主要是把后台从“能用”进一步升级为“能运营、能规模化管理”。

### 5.2.1 规则引擎 / 验证规则

#### 当前缺口

当前项目已经支持：

- 状态
- 过期时间
- 最大使用次数
- 套餐 / 产品映射

但还缺更成熟的规则能力，例如：

- 指定用户可用
- 指定分群可用
- 首单可用
- 指定来源渠道可用
- 指定活动时间窗可用
- 可叠加 / 不可叠加
- 自定义 metadata 条件

#### 为什么重要

因为一旦你开始做活动运营，就会很快出现：

- 新客专用码
- 老客续费码
- 淘宝渠道专用码
- 指定产品专用码
- 指定套餐升级码

没有规则层，这些能力都会退化成硬编码。

---

### 5.2.2 用户分群 / 目标用户运营能力

#### 当前缺口

当前项目更偏“码视角”，还没有形成强“用户分群视角”。

#### 建议补充

- 用户标签
- 来源渠道
- 历史兑换次数
- 历史订阅记录
- 当前资格状态
- 可发码对象筛选

#### 适用价值

这样你未来才能做：

- 给老客批量发续费码
- 给邀请成功用户批量发奖励码
- 给指定渠道新客发注册码

---

### 5.2.3 运营报表 / 渠道漏斗 / 对账

#### 当前缺口

你已经有操作历史，但还缺真正的运营报表。

#### 建议补充的指标

- 生成量
- 发放量
- 领取量
- 使用量
- 失效率
- 渠道转化率
- 活动转化率
- 批次消耗速度
- 退款撤销量
- 重复申领率

#### 为什么重要

因为你当前业务很依赖外部平台流量，运营上必须知道：

- 哪个平台核销效果更好
- 哪个商品链接更容易转化
- 哪个批次问题最多
- 哪些订单号经常有异常申领

---

## 5.3 P2：面向“产品授权平台 / License 平台”演进时必须补

这些能力不一定是你今天立刻最需要的，但如果未来你要把系统真正做成软件授权中心，就必须提前规划。

### 5.3.1 设备绑定 / 机器激活 / 心跳

#### 当前缺口

当前授权模型还偏“用户与订阅”，缺少“设备 / 机器”维度。

#### 成熟平台典型能力

- 机器指纹
- 激活台数限制
- 设备解绑
- 心跳检查
- 超时回收

#### 什么时候需要做

当你开始把它接入：

- 桌面软件
- 客户端程序
- 本地授权产品

时，这层就会变成核心。

---

### 5.3.2 离线授权 / 本地验签

#### 当前缺口

当前更适合在线场景，不适合离线激活或弱联网授权场景。

#### 典型能力

- 离线授权文件
- 本地验签
- 有效期签名
- 机器绑定签名

#### 适用场景

- 企业内网
- 不能稳定联网的软件
- 交付型桌面程序

---

### 5.3.3 细粒度 Entitlement（功能级授权）

#### 当前缺口

当前已经有产品资格与订阅能力，但粒度还不够细。

#### 建议演进方向

把权限拆得更细，例如：

- `novel_basic`
- `novel_pro`
- `desktop_activation`
- `offline_mode`
- `api_batch_export`
- `advanced_models_access`

#### 为什么需要

这样项目才能从“发一个订阅”进一步演进为“给一个产品/功能能力包发权”。

---

## 6. 当前最适合你业务现实的推荐路线

结合你当前的国内业务现实：

- 依赖淘宝 / 闲鱼等外部销售渠道
- 不稳定依赖在线支付回调
- 更适合人工审核 + 自动发权混合方式
- 目标是做“用户中心 + 授权中心 + 码中心”的结合体

当前最合理的路线不是先做复杂支付，而是：

## **先把“订单申领 + 审核发权 + 码发放 + 撤销回收 + 产品回调”这条闭环补齐。**

### 推荐顺序

#### 第一阶段

1. 订单申领页
2. 后台审核页
3. 审核通过后发订阅 / 发码
4. 站内发放记录

#### 第二阶段

5. 补发 / 撤销 / 回滚
6. 产品接入 Webhook
7. 批次 / 活动独立对象

#### 第三阶段

8. 规则引擎
9. 用户分群
10. 运营报表

#### 第四阶段

11. 设备绑定
12. 离线授权
13. 细粒度 Entitlement

---

## 7. 现阶段不建议优先投入的方向

以下能力虽然成熟平台常见，但不建议你现在优先做：

- 复杂支付渠道深度集成
- 多折扣叠加复杂结算
- 地理围栏 / 多地区税务细分
- POS / 线下零售整合
- 高级营销自动化编排
- 复杂 B2B 合同折扣模型

原因很简单：

你当前最核心的瓶颈不是“支付算得不够复杂”，而是：

> **外部购买结果如何可靠地回到站内，并且稳定发权、可追踪、可回收。**

---

## 8. 一句话结论

如果把当前项目和成熟码/授权平台做对比，那么：

### 当前项目已经解决了“码怎么建、怎么管、怎么查”的问题，

但还没有完全解决：

### **码怎么发、怎么审、怎么回、怎么通知产品、怎么形成闭环运营**。

因此，接下来最值得投入的不是再增加更多相似管理页，而是补下面这条链路：

## **模板/活动 → 生成码 → 发放/申领 → 审核发权 → 产品同步 → 撤销/补发/回收 → 报表审计**

这条链路补齐后，项目才会真正从“码管理后台”升级成“授权运营平台”。
