# 默认前端 vs 经典版：二开功能缺口矩阵（2026-05-02）

## 1. 范围

本清单仅针对当前你关心的二开主链路：

- `code-center`
- `code-publication`
- `order-claim`
- `order-claim-admin`
- `midjourney`
- 以及相关跨页跳转（`/console/*` 迁移残留）

对照对象：

- 新版：`web/default/src/features/*`
- 旧版：`web/classic/src/pages/*` + 对应 hooks/components
- 后端契约：`router/api-router.go` + `controller/*.go` + `model/*.go`

## 3. 新版相对旧版仍缺失的关键能力（未完成）

> 下面是“旧版可用、新版未等价迁移”的功能，不是接口 404 问题。

### 3.1 `code-publication` 缺口（高优先级）

旧版（`CodePublicationCenter`）具备：

- 记录详情抽屉（publication + order_claim + code_object + deliveries + operation_logs）
- 按条件过滤（状态、码类型、来源平台、订单号、产品、时间段等）
- delivery 状态流转（pending/delivered/claimed/used/revoked）
- reissue / revoke / rollback 的完整操作反馈与风险提示
- CSV 导出、attempt 链路查看、定位跳转（到码中心/申领管理）

新版当前仅有：

- 简化列表
- 基于 `publication_id` 的手动动作提交

结论：新版 `code-publication` 仍是“薄壳”，未达到旧版运营能力。

### 3.2 `code-center` 缺口（高优先级）

旧版（`CodeCenter`）具备：

- Tab 聚合：overview/redemption/registration/subscription
- 三类码表格的统一入口
- 与旧独立页联动定位

新版当前仅有：

- 统计卡片（且目前仅总数可靠）

结论：新版 `code-center` 尚未迁移“统一管理入口”能力。

### 3.3 `order-claim-admin` 缺口（高优先级）

旧版具备：

- 审核弹窗（approve/reject）
- 按 `grant_type` 动态配置发放参数
- 自动生成并复制发放码
- `claim_id + auto_open=1` 定位与自动打开审核

新版当前仅有：

- 列表查看
- 全局关键词过滤

结论：新版缺少审核主流程，业务闭环未迁移。

### 3.4 `order-claim` 缺口（中高优先级）

旧版具备：

- 完整申领字段、凭证图片、多条件筛选
- 状态标签与审核结果细节

新版当前：

- 提交字段已改对第一轮
- 但仍缺凭证上传/显示与更细筛选 UI

### 3.5 `midjourney` 缺口（中优先级）

旧版区分用户/管理员查询入口：

- 管理员：`/api/mj/`
- 普通用户：`/api/mj/self/`

新版当前固定调用 `/api/mj/`，并依赖路由权限控制；若后续权限策略变化，存在行为偏差风险。

## 4. 当前风险分级

### P0（必须优先收口）

1. `code-publication` 仍未迁移旧版核心运营能力（详情、状态流转、操作日志）
2. `order-claim-admin` 仍无审核闭环（approve/reject + 发放）

1. `code-center` 统一管理能力未迁移，仅剩统计壳
2. `order-claim` 凭证与细筛选能力缺失

### P2（可并行）

1. `midjourney` user/admin 路由分流策略与旧版保持一致
