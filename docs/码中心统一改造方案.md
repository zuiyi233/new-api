# 码中心统一改造方案

> 文档日期：2026-04-09  
> 适用范围：当前项目中的兑换码、注册码、订阅码后台与用户兑换入口  
> 目标：在**不推翻现有业务语义**的前提下，统一管理员操作体验、补齐批量能力、增强运营管理与审计能力

---

## 1. 背景与结论

当前项目已经具备三类码能力的基础雏形：

- 兑换码：用于用户已有账号后的额度类权益变更
- 注册码：用于注册前置校验或首次开通某产品资格
- 订阅码：用于将外部购买结果映射为站内订阅套餐发放

从业务语义上看，这三类码**并不相同**，不建议直接粗暴合并为一张通用表，也不建议为了“页面少”而牺牲语义清晰度。

但从管理员实际使用感受看，当前后台已经出现以下问题：

- 三个页面结构高度相似，但功能重复、体验割裂
- 管理能力普遍偏简陋
- 批量操作能力明显不足
- 缺少批次、活动、渠道、导入导出历史、操作审计等运营视角

### 最终建议

本项目应采用如下路线：

## **统一做“码中心”后台管理层，但保留三种码的业务语义模型**

即：

- UI 统一
- 批量能力统一
- 审计能力统一
- 筛选与导出统一
- 底层仍保留兑换码 / 注册码 / 订阅码三类模型和服务

## 2. 当前项目现状（基于本仓库代码核对）

本节为基于当前仓库代码直接核对得到的事实，不是推测。

### 2.1 当前已存在的后台页面

管理员侧目前已有三套页面：

- 兑换码管理
- 注册码管理
- 订阅码管理

页面入口已存在于侧边栏与相关设置项中。

### 2.2 当前已存在的基础能力

三类码当前已具备的能力并不完全一致，但总体已经具备如下基础：

- 列表页
- 搜索 / 筛选
- 单条启用 / 禁用 / 删除
- 单条编辑
- 码内容查看 / 复制
- 批量生成（注册码、订阅码）
- 使用记录弹窗（注册码、订阅码）
- 当前页导出（注册码、订阅码）
- 所选复制（注册码、订阅码）
- 用户侧统一输入“兑换码或订阅码”入口

### 2.3 当前明显存在的问题

#### 问题 A：后台存在“三页重复建设”

目前管理员要在三个相似页面之间来回切换：

- 页面结构相似
- 列表结构相似
- 操作按钮相似
- 业务字段只是在少数列上有区别

这会造成：

- 学习成本增加
- 运维效率下降
- 功能标准不一致
- 后续维护成本持续上升

#### 问题 B：批量能力不完整

目前已存在：

- 所选复制
- 当前页导出

但缺少管理员实际最常用的批量操作：

- 批量删除所选
- 批量启用
- 批量禁用
- 批量归档
- 批量延期
- 批量修改备注
- 批量修改产品资格 / 套餐映射
- 导出所选
- 导出当前筛选结果

#### 问题 C：缺少运营视角

目前后台更接近“技术 CRUD 页”，还不是成熟的运营后台。

典型缺失包括：

- 批次 / 活动（Batch / Campaign）
- 渠道来源
- 创建人
- 导入记录
- 导出记录
- 操作历史
- 外部订单关联
- 标签 / 业务元数据

#### 问题 D：使用记录不够产品化

现有注册码、订阅码已有使用记录弹窗，但仍可继续增强：

- 谁用了
- 什么时候用的
- 从哪个入口用的
- 成功还是失败
- 失败原因是什么
- 与哪个外部订单关联

#### 问题 E：三类码页面能力标准不一致

当前现象：

- 兑换码页面已有部分批量操作
- 注册码、订阅码页面的批量管理能力较弱
- 三套页面顶部动作区不统一

这会让管理员形成明显割裂感。

## 3. 产品定位与设计原则

### 3.1 产品定位

本项目中的“码系统”建议正式定位为：

## **统一码中心（Code Center）**

用于承接以下业务：

- 准入控制
- 产品资格发放
- 订阅套餐交付
- 额度补偿 / 充值补发
- 外部电商平台成交后的站内履约

### 3.2 三种码的正式语义

#### 兑换码

```text
兑换码 Redemption Code
= 用户已有账号后的二次权益变更码
```

典型用途：

- 增加 quota
- 补偿
- 续充
- 活动赠送

#### 注册码

```text
注册码 Registration Code
= 首次注册 / 首次激活 / 首次开通某产品资格的码
```

典型用途：

- 注册前置校验
- 新产品首开
- 渠道首购发权

#### 订阅码

```text
订阅码 Subscription Code
= 将外部购买结果映射为某个 Subscription Plan 的交付码
```

典型用途：

- 淘宝 / 闲鱼成交后的套餐发放
- 月卡 / 季卡 / 年卡交付
- 套餐续费与交付

### 3.3 设计原则

本次改造遵循以下原则：

1. **业务语义不混淆**
   - 兑换码、注册码、订阅码继续保留独立语义
2. **后台体验统一**
   - 管理员进入一个统一码中心即可完成管理
3. **先补运营能力，再谈抽象统一**
   - 优先解决批量、筛选、导入导出、审计等问题
4. **尽量复用现有代码结构**
   - 能在现有列表、Hook、Modal、路由基础上迭代，就不大拆
5. **对用户入口尽量统一，对管理员语义保持清晰**
   - 用户可以统一输入码
   - 管理员仍然清楚每类码的业务含义

## 4. 推荐的信息架构

### 4.1 现状信息架构

当前：

- 兑换码管理
- 注册码管理
- 订阅码管理

三者独立存在于后台导航中。

### 4.2 推荐信息架构

建议改造为：

## **码中心**

进入码中心后，使用 Tab / Segmented / 类型筛选切换：

- 全部
- 兑换码
- 注册码
- 订阅码

### 4.3 为什么不建议继续维持三套完全独立页

原因：

- 重复组件太多
- 操作按钮标准不统一
- 批量操作实现重复
- 后续新增“批次 / 导入导出 / 审计”会变成三份开发

### 4.4 为什么也不建议后端直接粗暴合并

原因：

- 三类码的业务语义不同
- 对应字段不同
- 对应履约逻辑不同
- 对应风控与校验逻辑不同

因此最合理的结构是：

## **UI 统一、能力统一、模型不强合并**

---

## 5. 必做改造项（P0）

这一阶段目标是：**先把后台从“能用”提升到“顺手可管理”**。

### 5.1 统一顶部操作区

建议三类码页面统一为同一套动作标准：

- 新建
- 批量复制
- 批量启用
- 批量禁用
- 批量删除 / 归档
- 导出
- 导入
- 刷新

### 5.2 批量删除所选

当前注册码、订阅码都缺少最基础的批量删除所选能力。

建议补齐：

- 批量删除所选注册码
- 批量删除所选订阅码
- 兑换码也统一支持批量删除所选

### 5.3 批量启用 / 禁用

单条点“更多”逐条操作过于低效。

建议新增：

- 批量启用
- 批量禁用

适用对象：

- 兑换码
- 注册码
- 订阅码

### 5.4 导出增强

当前注册码、订阅码仅支持“导出当前页”，不够用。

建议扩展为：

- 导出所选
- 导出当前筛选结果
- 导出全部
- 导出使用记录

### 5.5 失效码处理标准化

建议新增统一能力：

- 清理已过期
- 清理已用尽
- 清理已禁用
- 优先“归档”，不优先硬删除

### 5.6 使用记录导出

注册码、订阅码已有使用记录弹窗，建议统一支持：

- 导出当前码的使用记录
- 导出当前筛选范围内的使用记录

---

## 6. 运营化增强（P1）

这一阶段目标是：**把后台从 CRUD 页升级为真正的运营后台**。

### 6.1 增加批次 / 活动概念

建议至少增加如下字段：

- `batch_no`
- `campaign_name`
- `channel`
- `source_platform`
- `created_by`
- `notes`

适用场景：

- 淘宝活动
- 闲鱼发货
- 补偿发码
- 邀请奖励
- 节日活动
- 私域首购

### 6.2 高级筛选

建议新增更完整的筛选维度：

- 按码类型
- 按状态
- 按可用性
- 按产品资格
- 按订阅套餐
- 按批次
- 按渠道
- 按创建人
- 按时间范围
- 按是否已使用
- 按是否已用尽
- 按是否过期
- 按外部订单号

### 6.3 保存筛选视图

后台高频视图往往会被反复使用，建议支持：

- 保存当前筛选方案
- 默认视图
- 最近使用视图

典型示例：

- 本周已用尽订阅码
- 淘宝来源未使用注册码
- 本月补偿兑换码
- 即将过期订阅码

### 6.4 导入与导出历史

建议新增导入导出任务记录：

- 操作人
- 时间
- 文件名
- 数量
- 成功数 / 失败数
- 错误原因

### 6.5 使用记录增强

建议扩充 usage 记录维度：

- user_id
- used_at
- ip
- user_agent（可选）
- source_entry
- external_order_no
- result_status
- error_message

---

## 7. 模型增强建议（P2）

这一阶段目标是：**让码系统具备更强的产品化表达能力**。

### 7.1 标签与元数据能力

建议先做轻量版，不必一步到位做复杂 JSON Schema：

- `tags`
- `channel`
- `source_platform`
- `external_order_no`
- `batch_no`
- `notes`

后续如确实需要，再升级为正式 metadata 字段。

### 7.2 使用对象限制

建议未来预留更强限制条件：

- 指定用户可用
- 指定来源用户可用
- 首次注册可用
- 已有账号才能使用
- 限定某产品持有者可用

### 7.3 模板 / 批次 / 实例 三层结构

中期建议演进为：

#### 模板层

定义规则：

- 类型
- 默认产品资格 / plan_id / quota
- 默认有效期
- 默认备注
- 默认渠道

#### 批次层

定义活动：

- 批次名
- 渠道
- 来源
- 创建人
- 数量
- 导入方式

#### 实例层

定义单个码：

- code
- status
- expires_at
- used_count
- usage records

此结构适合未来向更完整的授权中心演进。

## 8. 用户侧建议

### 8.1 用户侧不建议继续拆更多页面

用户视角下最重要的是：

- 不需要理解三类码的内部区别
- 输入一个码，系统自动识别即可

### 8.2 推荐用户体验

统一入口继续保持：

```text
请输入兑换码或订阅码
系统自动识别并处理
```

中期可以进一步演进为：

```text
请输入交付码
系统自动识别：
- 是额度兑换码
- 还是订阅码
- 还是其他未来支持的交付码
```

### 8.3 注册码仍保留在注册流程中

注册码更适合保留在：

- 注册页
- 首次激活页

而不是强行并入充值页。

## 9. 推荐的分阶段落地路线

### 第一阶段：后台可用性补齐（优先级最高）

目标：

- 先让管理员不再觉得三页难用

改造项：

- 注册码 / 订阅码补批量删除所选
- 注册码 / 订阅码补批量启用 / 禁用
- 导出所选
- 统一顶部动作栏
- 使用记录支持导出
- 失效码支持归档

### 第二阶段：运营化能力增强

目标：

- 从 CRUD 页升级为运营后台

改造项：

- 新增批次字段
- CSV 导入
- 导入预校验
- 导入 / 导出历史
- 高级筛选
- 保存视图

### 第三阶段：码中心正式整合

目标：

- 形成统一码中心

改造项：

- 新增统一“码中心”入口
- 三类码共用一个列表框架
- 模板 / 批次 / 实例三层结构
- 更强的元数据与审计能力

## 10. 推荐的技术实现策略

### 10.1 前端层

建议复用并抽象当前现有结构：

- 列表页骨架
- 顶部 Actions 区
- Filters 区
- 使用记录 Modal
- 编辑 Modal
- 导出工具

目标是形成通用组件层，例如：

- `CodeCenterTableShell`
- `CodeCenterActions`
- `CodeCenterFilters`
- `CodeUsageModal`

### 10.2 后端层

建议在现有模型与接口基础上逐步新增：

- 批量删除接口
- 批量启用 / 禁用接口
- 批量导出接口
- 批量导入接口
- 批次字段支持
- 操作日志与导入导出记录接口

### 10.3 数据层

短期内不建议立刻把三类码合并成一张总表。

建议先做：

- 各自模型补字段
- 各自列表支持统一查询参数风格
- 统一操作日志表或统一导入导出记录表

## 11. 风险与注意事项

### 11.1 不建议一上来重构成“万能码表”

风险：

- 业务语义被冲淡
- 接口职责变复杂
- 容易造成兑换逻辑与注册逻辑串味

### 11.2 不建议一开始就做过度抽象

建议先补：

- 批量操作
- 导出导入
- 批次与筛选

等后台需求稳定后，再抽象通用层。

### 11.3 优先保留历史记录

后台数据建议优先：

- 归档
- 禁用
- 标记失效

不要把“删除”作为唯一清理手段。

## 13. 最终执行建议

本项目下一步不建议继续分散地补三个页面，而建议按如下顺序推进：

### 建议顺序

1. 先补齐 P0 批量能力
2. 再加 P1 运营能力
3. 最后整合成统一码中心

### 一句话总结

## **不要把三种码的业务语义硬合并，但一定要把后台管理体验统一。**

这才是当前项目最稳、最实用、最符合后续产品化方向的路线。
