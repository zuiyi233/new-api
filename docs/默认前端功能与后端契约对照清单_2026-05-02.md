## 3. 已确认的 API 错配清单

下面这些不是猜测，是我直接在前端 API 文件和后端 router/controller 里对照出来的。

| 功能页 | 默认前端调用 | 后端实际情况 | 结论 |
|---|---|---|---|
| `code-center` | `GET /api/code-center/stats` | 后端只看到 `GET /api/code-center/history`、`POST /api/code-center/history/export` | **明确 404 错配** |
| `code-publication` | `POST /api/code-publication/publish` | 后端没有这个 `publish` 入口；后端现有的是列表、详情、补发、撤回、回滚，以及 `code-delivery` 相关路由 | **明确 404 错配** |
| `order-claim` | `POST /api/order-claim/claim`、`GET /api/order-claim/my` | 后端实际是 `POST /api/order-claim/self`、`GET /api/order-claim/self` | **明确 404 错配** |
| `order-claim-admin` | `GET /api/order-claim/admin/search` | 后端没有 `search` 路由；只有 `GET /api/order-claim/admin/`、`GET /:id`、`POST /:id/review` | **明确 404 错配** |
| `midjourney` | `GET /api/mj/search` | 后端只有 `GET /api/mj/` 和 `GET /api/mj/self/` | **明确 404 错配** |

### 3.1 已对齐但要注意尾斜杠的接口

这些接口本身看起来是对得上的，但因为 Gin 路由有尾斜杠行为，后续如果继续报错，还是要看实际请求是否被重定向、是否被代理层改写：

- `GET /api/code-publication/?...`
- `GET /api/order-claim/admin/?...`
- `GET /api/mj/?...`

也就是说：**不是所有报错都来自“接口不存在”**，但上表里的 `publish`、`claim`、`my`、`search` 这些是可以直接定性为错配的。

## 4. 已确认的旧 `/console/*` 迁移残留

默认前端里还残留了几处经典版 `/console/*` 跳转。  
由于默认前端路由清单里没有 `/console` 根路由，这些跳转一旦触发，就会直接走到 404。

### 4.1 `code-publication`

文件：`N:\new-api-main\web\default\src\features\code-publication\index.tsx`

这里仍然硬编码了：

- `/console/subscription`
- `/console/code-center?...`
- `/console/order-claim-admin?...`

### 4.2 `channels`

文件：

- `N:\new-api-main\web\default\src\features\channels\components\channels-columns.tsx`
- `N:\new-api-main\web\default\src\features\channels\components\dialogs\channel-test-dialog.tsx`

这里仍然会跳：

- `/console/deployment?deployment_id=...`
- `/console/setting?tab=ratio`

### 4.3 `playground`

文件：`N:\new-api-main\web\default\src\features\playground\components\message-error.tsx`

这里也会打开：

- `/console/setting?tab=ratio`

### 4.4 这类问题的本质

这类不是后端接口不存在，而是**默认前端还在指向经典版页面路由**。  
如果这些入口在默认前端被点到，就会表现为：

- 直接 404
- 或者先跳转再报错

所以它和上面那类 API 错配是两条线，要分开看。

## 5. 这次已经能排除的判断

这次不应该再笼统地说成“后端没有接口，所以前端全炸了”。  
更准确的说法是：

1. **有些功能前端确实换了 API 路径，但没跟后端对齐。**
2. **有些功能后端其实有接口，只是前端还在调用旧路径。**
3. **有些导航错误来自默认前端残留的经典版 `/console/*` 路径。**
4. **500 不是静态扫描就能彻底定根因的，必须看真实响应体和后端日志。**
