# 统一用户与授权平台架构（B+C 混合模式）

> 适用对象：以当前 `new-api` 项目为底座，继续扩展“用户中心 + 产品授权中心 + 订单核销中心 + 邀请归因中心”的后续开发者、产品设计者与运营同学。
>
> 本文档目标：把当前讨论结果固化成正式方案文档，避免后续重复讨论“项目是否需要重做”“注册码/兑换码如何定位”“淘宝/闲鱼这类外部销售渠道如何接入”等问题。

---

## 0. 一句话结论

当前最推荐的方向不是重做一套新系统，也不是把本项目仅仅当成 AI 网关继续使用，而是：

**以当前 `new-api` 项目为底座，保留现有用户、邀请码、注册码、兑换码、权限、后台、账务、订阅与 API 网关能力，在其上补齐“订单申领 / 审核发权 / 产品接入”这几个连接层，把项目正式产品化为统一用户与授权平台。**

这套平台的定位是：

1. 用户中心
2. 产品授权中心
3. 订单核销中心
4. 邀请归因中心
5. AI 网关能力（保留为附属能力或其中一个产品能力）

---

## 1. 方案背景

### 1.1 当前项目已经具备的事实基础

从代码现状看，当前项目已经不是一个“只有 API 转发能力”的单点系统，而是已经具备以下基础能力：

- 用户注册 / 登录 / OAuth / 会话
- 管理后台
- 邀请体系
- 兑换码体系
- 注册码体系
- 产品资格（entitlement）体系
- 权限门禁
- 额度 / 账务 / 充值 / 订阅能力
- API 网关与上游模型聚合能力

因此，后续最合理的策略不是“重做”，而是：

> **在现有成熟能力之上补连接层与产品化能力。**

### 1.2 当前业务现实

当前现实条件包括：

- 需要一个统一账户体系来承接多个产品或未来多个产品
- 需要注册码 / 兑换码 / 邀请码这类运营工具
- 需要用户购买后自动或半自动获得资格
- 主要销售渠道可能来自淘宝、闲鱼、私域、代理等外部平台
- 这些平台未必提供适合当前场景的标准支付回调 / 官方 API 接入能力

这意味着平台设计不应把“自动支付回调”作为唯一前提，而应该优先建立：

- 用户识别
- 资格发放
- 订单核销
- 渠道归因
- 产品接入

## 2. 正式产品定位

本方案建议将当前项目重新定义为：

## **统一用户与授权平台**

它的职责不是“替代所有业务系统”，也不是“变成通用 ERP / 财务软件”，而是集中承担：

- 用户身份管理
- 产品资格管理
- 注册码 / 兑换码管理
- 订单申领与审核发权
- 邀请码 / 渠道归因
- 产品授权结果输出

同时保留当前项目已有的 API 网关能力，作为：

- 平台的附属能力，或
- 某一类产品能力的承载层

## 4. 模块边界

本章重点回答：**哪些模块已经具备，哪些模块只需要补一层连接能力。**

### 4.1 用户中心

#### 当前已具备

- 用户注册 / 登录
- OAuth
- 会话与身份
- 管理后台用户管理

#### 模块职责

- 用户是谁
- 用户如何登录
- 用户和邀请码 / 注册码 / 资格之间的关联

#### 结论

**用户中心不需要重做，只需要继续作为平台底座保留。**

---

### 4.2 授权中心

#### 当前已具备

- 注册码体系
- 注册码使用记录
- 产品资格（entitlement）体系
- 权限门禁
- 用户自查资格接口

#### 模块职责

- 用户是否拥有某个产品资格
- 资格从哪里来
- 资格是否有效 / 是否到期
- 是否允许访问对应功能

#### 结论

**授权中心已经形成闭环，是当前项目最值得继续扩展的核心模块。**

---

### 4.3 邀请归因中心

#### 当前已具备

- 邀请体系

#### 建议升级方向

将邀请码扩展为两层语义：

1. 用户邀请：老用户邀请新用户
2. 渠道邀请码：淘宝店 / 闲鱼店 / 代理 / KOL / 社群入口归因

#### 模块职责

- 用户来源绑定
- 渠道来源绑定
- 后续统计与返利归因

#### 结论

**邀请码不要只作为“拉新工具”，应升级为平台级“来源与渠道归因工具”。**

---

### 4.4 注册码中心

#### 当前已具备

- 注册码管理后台
- 注册码校验接口
- 注册事务内二次校验
- 使用审计
- 成功注册后自动发放 entitlement

#### 建议正式语义

```text
注册码 Registration Code
= 用于“首次注册 / 首次激活 / 首次开通某产品资格”的码
```

#### 适合业务场景

- 产品首次开通
- 会员首次激活
- 私域或渠道首购发权
- 注册时直接授予产品资格

#### 结论

**注册码中心已可用，不需要重做，只需要加强产品化字段与运营管理能力。**

---

### 4.5 兑换码中心

#### 当前已具备

- 兑换码管理后台
- 用户兑换入口
- 给用户增加 quota 的事务逻辑
- 已使用 / 过期 / 状态控制

#### 建议正式语义

```text
兑换码 Redemption Code
= 用于“用户已有账号后的二次权益变更”
```

#### 适合业务场景

- 增加额度
- 加次数
- 续费
- 升级套餐
- 补偿用户

#### 建议补充：订阅码 / 套餐码

在当前项目中，现有兑换码更偏向：

- 钱包额度
- quota 增加
- 二次充值

但如果业务上需要承接：

- 月卡 / 季卡 / 年卡
- 有效期套餐
- 周期重置额度
- 分组升级
- 订阅优先扣费

则不建议把这类语义继续压进当前 `Redemption` 模型，而建议新增一类独立的：

```text
订阅码 / 套餐码 Subscription Code
= 用于“根据 plan_id 创建 UserSubscription”的交付码
```

其核心思路是：

- 外部平台负责成交
- 平台负责发码
- 用户回到站内输入订阅码
- 系统自动识别该码对应的是某个订阅套餐
- 后端直接复用现有订阅逻辑创建 `UserSubscription`

这条路线的关键价值在于：

- 不破坏现有订阅系统语义
- 不把订阅错误压扁成 quota 充值
- 能把“外部购买”和“站内订阅体系”正确连接起来

#### 统一输码原则（建议）

前台不必强迫用户区分“这是兑换码还是订阅码”。

更推荐的产品形态是：

- 用户看到统一的输码入口
- 输入任意交付码
- 系统自动识别其类型
- 根据类型执行不同后端逻辑

例如：

- 若识别为 `redemption code`，则执行额度充值
- 若识别为 `subscription code`，则创建用户订阅
- 若识别为 `registration code`，则在注册流程中使用

这意味着后续前端体验可以是：

- 兑换页可输入统一码
- 订阅页也可输入统一码
- 用户无需手工判断“我该去哪个页面输入”

#### 结论

**兑换码应继续保留为额度类交付工具；若业务涉及套餐、会员和有效期订阅，则建议新增“订阅码 / 套餐码”与之分工，而不是直接拿现有兑换码替代整个订阅语义。**

---

### 4.6 订单核销中心

#### 当前不足

当前项目已有支付与充值能力，但现实业务接入上仍存在一个明显缺口：

- 淘宝 / 闲鱼等外部销售渠道未必能提供适合当前场景的自动支付回调
- 订单结果无法稳定、标准化地直接进入本系统

#### 建议新增

新增：

- 订单申领页
- 后台审核页
- 订单申领表
- 发权记录表

#### 模块职责

- 承接外部平台订单结果
- 提交订单号 / 截图 / 联系方式 / 平台信息
- 后台审核
- 审核通过后发注册码 / 发兑换码 / 直接发权

#### 结论

**订单核销中心是本方案中最值得新增的连接层，它不是支付系统，而是“外部订单进入平台后的标准入口”。**

---

### 4.7 产品接入层

#### 目标

你的业务产品不再自己维护一套授权逻辑，而是直接消费主平台输出的身份与资格结果。

#### 推荐接入路径

##### 方式 A：查询式接入

产品调用主平台接口：

- 当前用户是谁
- 拥有哪些资格
- 是否有效
- 套餐或额度状态

##### 方式 B：Token 式接入

主平台签发带 entitlement 的 token，由业务产品本地校验。

##### 方式 C：SSO 式接入

未来多产品矩阵时再做统一登录与统一授权同步。

#### 当前建议

**优先做查询式接入，再逐步演进到 Token 式接入。**

## 5. 核心业务流

### 5.1 首次开通：注册码流

```text
用户购买商品
   ↓
获得注册码
   ↓
访问主平台注册页
   ↓
填写邀请码（可选/渠道化）
   ↓
填写注册码
   ↓
系统注册成功
   ↓
自动发放产品资格
   ↓
产品端读取资格并放行
```

### 5.2 二次增购：兑换码流

```text
用户已有账号
   ↓
购买加量/续费商品
   ↓
获得兑换码
   ↓
登录主平台兑换
   ↓
系统增加额度 / 延长权益 / 升级套餐
   ↓
产品端读取最新资格或额度
```

### 5.3 外部购买订阅：订阅码 / 套餐码流

```text
用户在淘宝/闲鱼购买订阅商品
   ↓
外部平台自动发货或人工发放订阅码
   ↓
用户回到站内订阅页或统一输码入口
   ↓
输入订阅码
   ↓
系统自动识别该码对应的 plan_id
   ↓
后端创建 UserSubscription
   ↓
订阅立即生效（有效期 / 总额度 / 重置周期 / 分组升级按 plan 执行）
   ↓
产品端按订阅状态与 billing preference 生效
```

### 5.4 外部渠道订单：订单申领 + 审核发权流（兜底）

```text
用户在淘宝/闲鱼下单
   ↓
进入主平台“订单申领”
   ↓
提交订单号 / 截图 / 平台信息
   ↓
后台审核
   ↓
审核通过后执行：
   ├─ 发注册码
   ├─ 发兑换码
   └─ 或直接发 entitlement
   ↓
用户获得资格
   ↓
产品端自动识别权限
```

### 5.5 邀请 / 渠道归因流

```text
渠道邀请码 / 邀请链接
   ↓
用户通过特定入口注册
   ↓
系统绑定来源关系
   ↓
后续购买 / 激活 / 续费都带上归因
   ↓
后台统计渠道效果 / 返佣 / 代理贡献
```

## 6. 数据模型建议

本方案强调：**尽量不推翻现有模型，只补最少的新模型来承接外部订单与发权记录。**

### 6.1 当前可直接复用的核心模型

- `User`
- 邀请相关模型
- `RegistrationCode`
- `RegistrationCodeUsage`
- `Redemption`
- `UserProductEntitlement`
- 已有 topup / payment / subscription 等账务能力

### 6.2 建议新增模型 1：`subscription_codes`

作用：

- 作为“订阅套餐”的交付码
- 连接外部购买结果与现有 `SubscriptionPlan` / `UserSubscription`
- 不改变现有订阅模型，只增加一种交付入口

建议字段：

```text
id
code
plan_id
status                 // enabled / disabled / used / expired
expires_at
max_uses
used_count
created_by
notes
created_at
updated_at
```

建议配套 usage 表：

```text
subscription_code_usages
  -> id
  -> subscription_code_id
  -> user_id
  -> used_at
  -> ip
  -> notes
```

建议消费逻辑：

- 用户输入订阅码
- 根据 `plan_id` 查到对应套餐
- 复用现有订阅创建逻辑：
  - `AdminBindSubscription(...)`
  - 或 `CreateUserSubscriptionFromPlanTx(...)`

### 6.3 建议新增模型 2：`order_claims`

作用：

- 记录用户提交的外部订单申领
- 作为人工审核和核销入口

建议字段：

```text
id
user_id
platform               // taobao / xianyu / other
external_order_no
product_key
claim_type             // registration / redemption / direct_grant
status                 // pending / approved / rejected / completed
snapshot_urls          // 支付截图 / 订单截图
contact_info
invite_code
channel_code
notes
reviewed_by
reviewed_at
created_at
updated_at
```

### 6.4 建议新增模型 3：`order_grants`

作用：

- 记录订单申领最终通过何种方式发权

建议字段：

```text
id
order_claim_id
grant_type             // registration_code / redemption_code / direct_entitlement
registration_code_id
redemption_id
entitlement_id
granted_to_user_id
granted_by
notes
created_at
```

### 6.5 建议新增统一输码接口（可选但推荐）

为了让用户不必区分“兑换码”和“订阅码”，建议后续新增统一输码接口，例如：

```text
POST /api/user/redeem-code
```

建议返回结构：

```json
{
  "success": true,
  "data": {
    "code_type": "redemption | subscription | registration",
    "action": "quota_topup | create_subscription | register_only",
    "message": "兑换成功",
    "subscription": {},
    "quota": 100000
  }
}
```

建议处理原则：

- 先查订阅码
- 再查兑换码
- 注册码仍保留注册流程专用校验与消费逻辑
- 前端统一展示结果，不要求用户自行判断类型

---

## 7. 与支付中心的关系

### 7.1 本方案不否定现有支付能力

当前项目已有支付、充值、订阅等能力，这些都应保留。

### 7.2 但不应把“自动支付回调”当成唯一前提

在淘宝 / 闲鱼 / 私域等现实场景下，可能存在：

- 无法稳定获取标准支付回调
- 无法直接与本系统做自动订单同步
- 无法完全依赖平台开放能力

因此，本方案的关键判断是：

> **现有支付中心继续保留，但在业务落地上，应以“外部购买 + 发码交付”为主路径，以“订单申领 + 审核发权”为兜底路径。**

### 7.3 推荐理解

对当前业务最实用的定义不是：

- “支付中心 = 收款网关”

而是：

- “支付与订单结果进入系统后，平台负责核销、发码、发权和审计”

### 7.4 与外部自动发货的配合方式

若外部销售渠道支持自动发货，则推荐：

- 额度类商品：自动发 `兑换码`
- 订阅类商品：自动发 `订阅码 / 套餐码`

用户回到站内后，只需要输入交付码，系统自动识别并完成：

- 加额度
- 创建订阅
- 更新权限结果

这样可以避免用户在站内频繁切换“订阅页 / 兑换页 / 审核页”。

## 8. 分阶段落地建议

### Phase 1：先把授权链路做扎实

目标：

- 注册码首次开通
- 兑换码续费 / 加量
- entitlement 查询稳定
- 产品端可读取资格结果
- 邀请与渠道归因先保留基础能力

### Phase 2：补齐订单核销中心

目标：

- 用户侧订单申领页
- 管理侧审核发权页
- 订单申领与发权记录表
- 审计闭环

### Phase 3：增强渠道与代理能力

目标：

- 邀请码升级为渠道邀请码
- 统计不同渠道成交与转化
- 为返佣 / 代理管理预留空间

### Phase 4：再考虑支付自动化

目标：

- 在未来具备更合适的支付渠道条件时，再把自动支付回调接入做强

注意：

**支付自动化不是本方案的前置条件。**

## 10. 风险与限制

### 10.1 当前不是完整 License Server

虽然当前项目已经具备“注册码 + 资格 + 门禁”的能力，但它仍然不等于完整桌面 License Server。

当前方案更适合：

- 用户账号授权
- 产品资格发放
- 会员 / 权益 / 额度控制

而不是原生支持：

- 设备绑定
- 机器码
- 离线激活
- 授权文件签发
- 多终端激活策略

### 10.2 订单核销流程可能先是半自动

在外部销售渠道开放能力不足的情况下，订单申领与审核发权可能先以半自动为主。

这不是缺陷，而是现实条件下更稳的落地方式。

### 10.3 需要补一定的自动化测试

建议后续补齐至少以下测试：

- 注册码有效 / 无效 / 过期 / 耗尽
- 兑换码重复兑换 / 并发兑换
- entitlement 发放逻辑
- 订单申领审核后的发权逻辑

## 11. 最终实施原则

### 原则 1：不重做，只加连接层

保留当前项目已经成熟的用户、授权、后台、账务与 API 网关能力。

### 原则 2：注册码与兑换码分工明确

- 注册码：首次开通
- 兑换码：续费 / 增购 / 补偿

### 原则 2.1：订阅不压扁成额度充值

若业务卖的是：

- 月卡 / 季卡 / 年卡
- 有效期套餐
- 周期重置额度
- 订阅优先扣费能力

则应使用“订阅码 / 套餐码”接入现有订阅系统，而不是简单压成当前 quota 兑换码。

### 原则 3：支付不强求自动回调闭环

优先使用“外部购买 + 发码交付”，在必要时再以订单申领 + 审核发权承接兜底闭环。

### 原则 4：产品不自己做授权中心

所有授权都由主平台统一管理，业务产品只消费授权结果。

### 原则 5：前台统一输码，后台自动识别

对用户来说，应尽量减少“页面切换”和“码类型判断”负担。

推荐实现：

- 订阅页支持输入交付码
- 兑换页支持输入交付码
- 或统一收口成一个输码入口

后台自动识别：

- 兑换码 → 加额度
- 订阅码 → 开订阅
- 注册码 → 仅用于注册流程

## 13. 文档结论

本方案的核心不是“把当前项目改成另一个完全不同的系统”，而是：

> **把当前项目已经具备的成熟能力正式组织起来，补齐订单核销与产品接入层，形成一套可长期演进的统一用户与授权平台。**

对后续开发而言，最重要的不是推翻重做，而是坚持以下判断：

- 当前项目足够完整，适合作为基座
- 注册码 / 兑换码 / 邀请码已经具备业务基础
- entitlement 体系已经是平台化授权能力的核心
- “订阅码 / 套餐码”是连接外部购买与现有订阅体系的最关键新增连接层
- 订单申领 / 审核发权继续作为兜底路径保留
- 前台应尽量采用统一输码、自动识别的体验设计
- 你的业务产品应只消费授权结果，而不再各自维护独立授权逻辑
