## 1. 一句话目标

在**不重做当前项目**的前提下，以现有用户、订阅、资格、码中心和后台能力为底座，按阶段补齐：

## **订单申领 → 审核发权 → 发放追踪 → 产品同步 → 撤销回收 → 运营报表**

最终把项目从“码管理后台”推进为“统一用户与授权平台”。

---

## 2. 背景

当前项目已经具备：

- 用户中心
- 订阅套餐与用户订阅
- 产品资格 / entitlement 雏形
- 兑换码 / 注册码 / 订阅码
- 邀请体系
- 码中心运营后台基础能力
- AI 网关能力

但目前系统仍主要解决了：

- 码怎么建
- 码怎么管
- 码怎么查

还没有完全解决：

- 外部购买如何回站内
- 用户如何申领
- 管理员如何审核
- 如何稳定发权
- 如何撤销 / 补发 / 回收
- 如何通知产品侧同步生效

---

## 3. 问题陈述

当前项目与成熟码/授权平台相比，最关键的缺口不在“再多几个后台页面”，而在于缺少完整业务闭环。

核心缺口包括：

1. 缺少正式的**发放层 / 发布层**
2. 缺少**订单申领 / 审核发权**
3. 缺少**撤销 / 补发 / 回滚**
4. 缺少**产品接入事件 / Webhook**
5. 缺少**模板 / 活动 / 批次正式化**
6. 缺少**规则引擎 / 用户分群 / 报表**
7. 缺少面向软件授权的**机器绑定 / 离线授权 / 细粒度 entitlement**

---

## 4. 总体策略

本阶段计划遵循以下原则：

### 4.1 不重做现有系统

继续复用现有：

- 用户模型
- 订阅体系
- entitlement 体系
- 三类码模型
- 后台管理框架

### 4.2 先补连接层，再补高级能力

优先补齐：

- 外部订单进入系统
- 审核与发权
- 发放追踪
- 撤销回收
- 产品同步

之后再做：

- 规则
- 报表
- 设备授权

### 4.3 保持语义清晰

不把所有业务都压成一张“万能码表”，而是：

- 保留兑换码 / 注册码 / 订阅码语义
- 在其上增加统一运营层

### 4.4 按 Trellis 的跨层思路推进

每一阶段都明确：

- 数据流
- 边界契约
- 最小交付
- 验收标准

---

## 5. Cross-Layer 总数据流

在 Trellis 的 cross-layer 思路下，本项目后续最关键的业务流应先统一抽象为：

```text
外部购买
  -> 用户回站内提交订单申领
  -> 后台审核
  -> 发码 / 发订阅 / 发资格
  -> 记录发放轨迹
  -> 产品侧接收事件
  -> 用户实际生效
  -> 支持撤销 / 补发 / 回收
  -> 进入报表与审计
```

### 关键边界

#### 边界 A：前端申领页 -> 后端 API

输入：

```json
{
  "source_platform": "taobao",
  "external_order_no": "TB202604090001",
  "buyer_contact": "user@example.com",
  "proof_images": ["https://..."],
  "claimed_product": "subscription_pro"
}
```

输出：

```json
{
  "success": true,
  "data": {
    "claim_id": 1001,
    "status": "pending_review"
  }
}
```

#### 边界 B：后台审核 -> 发权服务

输入：

- 审核结论
- 发权方式
- 目标产品 / 套餐 / 资格

输出：

- 发权记录
- 码发放记录或订阅记录
- 审计日志

#### 边界 C：授权中心 -> 产品侧

输入：

- 发权事件
- 撤销事件
- 回收事件

输出：

- 产品同步成功 / 失败状态
- 重试记录

#### 边界 D：运营后台 -> 报表层

输入：

- 申领记录
- 审核记录
- 发放记录
- 使用记录
- 回收记录

输出：

- 渠道统计
- 活动转化
- 批次表现
- 异常对账

## 6. 分阶段计划

---

## Phase 1：订单申领最小闭环

### 6.1 目标

建立一条**不依赖在线支付回调**、但可以正式承接淘宝 / 闲鱼 / 私域订单的最小闭环。

### 6.2 交付内容

#### 后端

- 新增 `order_claims`
- 新增申领状态机
- 新增用户提交申领接口
- 新增后台审核接口
- 审核通过后支持：
  - 发订阅
  - 发订阅码
  - 发兑换码
  - 发注册码

#### 前端

- 用户侧订单申领页
- 后台订单申领列表页
- 后台审核弹窗 / 审核页

#### 数据/审计

- 记录申领人
- 记录平台来源
- 记录外部订单号
- 记录审核人
- 记录审核备注

### 6.3 非目标

- 不做淘宝/闲鱼官方 API 自动拉单
- 不做复杂自动风控
- 不做设备授权

### 6.4 验收标准

- 用户可提交订单申领
- 后台可审核通过 / 驳回
- 审核通过后能成功发出目标权益
- 全链路有基本日志与审计记录

---

## Phase 2：发放层 / 发布层正式化

### 6.5 目标

解决“码已经生成，但到底发给谁、是否已送达、是否已领取”的问题。

### 6.6 交付内容

#### 后端

- 新增 `code_publications` / `code_deliveries`（或同等语义模型）
- 支持码绑定：
  - 用户
  - 订单
  - 联系方式
  - 渠道
- 支持发放状态：
  - 待发放
  - 已发放
  - 已送达
  - 已领取
  - 已使用
  - 已撤回

#### 前端

- 后台发放记录页
- 订单申领详情中的发放记录区
- 批次与发放记录联动查看

### 6.7 验收标准

- 任一发出的码都能查到发放记录
- 同一外部订单号可以定位发放结果
- 能区分“已发出”和“已使用”

---

## Phase 3：撤销 / 补发 / 回收 / 回滚

### 6.8 目标

把“disabled”升级为真正业务语义上的生命周期控制。

### 6.9 交付内容

#### 后端

- 新增撤销原因
- 新增补发逻辑
- 新增回收 / 回滚服务
- 支持退款后回收订阅或资格
- 支持旧码作废、新码补发

#### 前端

- 发放记录页补“撤销 / 补发”操作
- 审核页支持撤销已发权结果

### 6.10 验收标准

- 已发放权益可以撤销
- 已发错权益可以补发
- 有审计记录和操作历史

---

## Phase 4：产品接入 / Webhook / 同步状态

### 6.11 目标

让后台发权结果不只停留在管理系统里，而是能同步到产品侧。

### 6.12 交付内容

#### 后端

- 事件表
- Webhook 配置
- Webhook 重试
- 基础签名验证
- 事件查询接口

#### 事件类型

- `claim.approved`
- `claim.rejected`
- `subscription.granted`
- `subscription.revoked`
- `code.redeemed`
- `entitlement.granted`
- `entitlement.revoked`

#### 前端

- 产品接入配置页
- Webhook 投递日志页

### 6.13 验收标准

- 发权后可触发产品同步事件
- 投递失败可重试
- 可查看事件日志

---

## Phase 5：模板 / 活动 / 批次正式化

### 6.14 目标

把当前批次字段升级为真正的运营对象。

### 6.15 交付内容

#### 后端

- `code_templates`
- `code_campaigns`
- `code_batches`
- 码实例与批次关联

#### 前端

- 模板管理页
- 活动管理页
- 批次详情页
- 活动 / 批次统计页

### 6.16 验收标准

- 能从模板派生批次
- 能查看批次下码实例情况
- 能按活动维度做统计

---

## Phase 6：规则引擎 / 用户分群 / 报表

### 6.17 目标

把后台从“能发码”升级为“能运营”。

### 6.18 交付内容

#### 规则引擎

- 首单规则
- 指定渠道规则
- 指定产品规则
- 可叠加 / 不可叠加
- 时间窗规则
- metadata 条件

#### 用户分群

- 用户标签
- 来源渠道
- 历史购买 / 历史兑换
- 当前资格状态

#### 报表

- 生成量 / 发放量 / 使用量
- 渠道转化
- 活动表现
- 批次消耗
- 重复申领率
- 异常订单分析

### 6.19 验收标准

- 运营可以按规则配置活动
- 可以按用户群体投放不同码
- 可以看到活动与渠道的基本效果

---

## Phase 7：License 平台能力（可选长期）

### 6.20 目标

如果未来要把平台扩展为软件授权中心，则进入此阶段。

### 6.21 交付内容

- 机器绑定
- 激活台数限制
- 心跳检查
- 离线授权文件
- 本地验签
- 细粒度 entitlement

### 6.22 验收标准

- 支持在线与离线授权
- 支持设备维度限制
- 支持功能级授权

---

#### 第二梯队

6. Phase 6：规则 / 分群 / 报表

## 8. 风险与依赖

### 8.1 主要风险

- 当前项目已有多套码模型，抽象时需要避免语义混乱
- 订单申领与发放层一旦落地，会影响多页面联动
- Webhook 涉及产品侧契约，需先定义最小协议
- 后续如果做设备授权，模型复杂度会明显上升

### 8.2 关键依赖

- 现有用户体系
- 现有订阅与 entitlement 模型
- 现有码中心后台
- 当前操作历史体系

---

## 9. 推荐的下一步

如果按 Trellis 的“先明确边界，再推进实现”方式继续，最推荐的下一步是：

## **把 Phase 1 单独拆成一份 PRD，并开始做“订单申领 + 审核发权”任务。**

原因：

- 它最贴近当前业务现实
- 它能绕过在线支付回调限制
- 它能把码中心、订阅、授权真正连起来
- 它是后续发放层、撤销回收、Webhook 的前置基础

---

## 10. 一句话结论

这份阶段计划的核心不是“继续堆更多管理页”，而是按 Trellis 的方式，把系统分阶段推进为：

## **统一用户与授权平台**

其最短闭环路线为：

## **订单申领 → 审核发权 → 发放追踪 → 产品同步 → 撤销回收**

只要这五步补齐，当前项目就会从“有码中心的后台”升级成真正能承接业务的授权运营平台。
