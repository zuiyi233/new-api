## 二、认证中间件体系

### 认证模式对照表

| 中间件 | 认证方式 | 说明 |
|--------|---------|------|
| `UserAuth()` | Session | 普通用户及以上 |
| `AdminAuth()` | Session | 管理员及以上 |
| `RootAuth()` | Session | 仅 Root 用户 |
| `TokenAuth()` | API Key（Bearer Token） | AI API 中继核心认证 |
| `TokenAuthReadOnly()` | API Key（宽松） | 不检查令牌状态/过期/额度 |
| `TryUserAuth()` | Session（可选） | 尝试认证，不强制 |
| `AccessTokenOnlyUserAuth()` | 系统访问令牌 | 跨站点中控场景 |
| `TokenOrUserAuth()` | Session 或 API Token | 二选一认证 |
| `OIDCAdminTokenAuth()` | OIDC 管理令牌 | OIDC 客户端管理 |
| `SecureVerificationRequired()` | 二次确认 | 敏感操作 5 分钟窗口 |

### TokenAuth 支持的密钥格式

| 格式 | 提取方式 | 说明 |
|------|---------|------|
| `Authorization: Bearer sk-xxx` | 标准方式 | OpenAI 兼容 |
| `x-api-key: sk-xxx` | Anthropic 格式 | Claude API 兼容 |
| `?key=sk-xxx` 或 `x-goog-api-key` | Gemini 格式 | **密钥出现在 URL 中，有泄露风险** |
| `mj-api-secret: xxx` | Midjourney 格式 | MJ 代理兼容 |
| `Sec-WebSocket-Protocol: openai-insecure-api-key.sk-xxx` | WebSocket | Realtime API |

### 中间件链路

**Relay 接口（核心 API 代理链路）：**
```
CORS → DecompressRequestMiddleware → BodyStorageCleanup → StatsMiddleware
  → RouteTag("relay") → SystemPerformanceCheck → TokenAuth
  → RequireProductEntitlement → ModelRequestRateLimit → UserConcurrencyLimit
  → Distribute → Controller
```

**API 管理接口链路：**
```
RouteTag("api") → BodyStorageCleanup → GlobalAPIRateLimit
  → [UserAuth/AdminAuth/RootAuth] → Controller
```

**Web 前端链路：**
```
gzip.Gzip → GlobalWebRateLimit → Cache → static.Serve
```

## 三、安全风险评估

### 🔴 严重风险

#### S1. CORS 配置允许任意来源跨域请求 + 携带凭据

**位置**：`middleware/cors.go`

```go
config.AllowAllOrigins = true
config.AllowCredentials = true
config.AllowHeaders = []string{"*"}
```

**影响**：任何恶意网站都可以携带用户的 Session Cookie 向本系统发起跨域请求，等同于 CSRF 攻击。攻击者可以在恶意网页中嵌入 JavaScript，以用户身份执行任意 API 操作（如创建令牌、修改密码、转移额度等）。

**在 CF 保护下的风险**：CF 的 WAF 和 DDoS 防护**无法防御**此类攻击，因为请求是从用户浏览器发出的合法请求，且经过 CF 代理。

---

#### S2. `/api/status` 端点无认证泄露大量系统信息

**位置**：`controller/misc.go:42-168`

泄露信息包括：

- 服务器版本号
- GitHub/Discord/LinuxDO OAuth Client ID
- Turnstile Site Key
- OIDC Client ID 和授权端点
- Passkey RP ID 和 Origins
- 服务器地址（`server_address`）
- 系统名称、邮箱验证状态、USD 汇率
- 自定义 OAuth 提供商信息

**影响**：攻击者无需任何认证即可获取系统指纹和配置信息，为后续攻击（如 OAuth 劫持、Passkey 欺骗）提供情报。

---

#### S3. 服务器绑定 `0.0.0.0` 且无 Cloudflare IP 验证

**位置**：`main.go:207-215`

```go
server.Run(":" + port)  // 等价于 0.0.0.0:port
```

关键问题：

1. 服务器始终绑定所有网络接口，**无法配置为仅监听 127.0.0.1**
2. 没有验证请求是否来自 Cloudflare 代理 — 未检查 `CF-Connecting-IP` 头部
3. 没有配置 Gin 的 `TrustedProxies` — `c.ClientIP()` 信任 `X-Forwarded-For` 头部，可被伪造
4. 没有全局 IP 白名单 — 无法限制只有 CF IP 才能访问

**影响**：如果攻击者发现了服务器的真实 IP 地址（通过 DNS 历史、子域名扫描、SSL 证书搜索等），可以直接绕过 Cloudflare 访问服务器。此时：

- 所有基于 IP 的限流都可被绕过（因为 `ClientIP()` 可伪造）
- Token 的 IP 白名单限制可被绕过
- CF 的 WAF 和 DDoS 防护完全失效

---

### 🟠 高风险

#### H1. pprof 调试端口无认证暴露

**位置**：`main.go:156-162`

```go
if os.Getenv("ENABLE_PPROF") == "true" {
    gopool.Go(func() {
        log.Println(http.ListenAndServe("0.0.0.0:8005", nil))
    })
}
```

**影响**：如果 `ENABLE_PPROF=true`，攻击者可直接访问 `http://IP:8005/debug/pprof/` 获取：

- 完整的 goroutine 堆栈（含函数名和代码路径）
- 堆内存数据（可能包含 API 密钥、Token 等敏感信息）
- CPU 性能分析数据

即使通过 CF 代理，如果攻击者发现了真实 IP 和 8005 端口，pprof 端口将完全暴露。

---

#### H2. Webhook 端点无认证保护

**位置**：`router/api-router.go:50-53`

```go
apiRouter.POST("/stripe/webhook", controller.StripeWebhook)
apiRouter.POST("/creem/webhook", controller.CreemWebhook)
apiRouter.POST("/waffo/webhook", controller.WaffoWebhook)
```

**影响**：虽然内部有签名验证，但如果签名密钥泄露或验证逻辑有缺陷，攻击者可以伪造支付成功通知，给自己充值额度。

---

#### H3. 大部分限流器基于 IP 维度，代理轮换可绕过

**位置**：`middleware/rate-limit.go`

除 `SearchRateLimit` 外，所有限流器都基于客户端 IP。在 CF 保护下，`ClientIP()` 获取的是 CF 传递的 IP，但如果攻击者直接访问真实 IP 并伪造 `X-Forwarded-For`，限流将完全失效。

---

#### H4. 错误信息直接返回客户端，泄露内部信息

**位置**：多处 controller 代码

大量 `err.Error()` 直接返回给客户端，可能泄露：

- 数据库错误信息（表名、列名、连接信息）
- 内部文件路径
- 上游 API 错误详情

例如 `controller/setup.go:109`：

```go
"message": "系统错误: " + err.Error()
```

---

#### H5. WebSocket CheckOrigin 始终返回 true

**位置**：`controller/relay.go:244-249`

```go
var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool {
        return true
    },
}
```

**影响**：存在跨站 WebSocket 劫持（CSWSH）风险，恶意网站可以建立 WebSocket 连接到 `/v1/realtime`。

## 四、控制器安全审计

### 1. SQL 注入风险

**整体评估：低风险**

- 业务代码大量使用 GORM 参数化查询，SQL 注入风险很低
- `model/main.go` 迁移代码中有 `fmt.Sprintf` 拼接 SQL，但表名/列名均为硬编码常量，不来自用户输入
- `model/user.go` 的 `SearchUsers` 使用 GORM `?` 占位符，安全

### 2. 认证绕过可能性

| 端点 | 风险 | 说明 |
|------|------|------|
| `/api/stripe/webhook` 等 | 🟠 高 | 无认证，依赖签名验证 |
| `/api/user/epay/notify` | 🟠 高 | 无认证，依赖签名验证 |
| `/api/subscription/epay/*` | 🟠 高 | 无认证，依赖签名验证 |
| `/api/setup` | 🟡 中 | 仅 `constant.Setup` 标志位防重复，首次部署可被抢注 |
| `/api/hub/session/bootstrap` | 🟡 中 | 无认证中间件保护 |
| `WssAuth()` 空实现 | 🟡 中 | 若被路由使用将导致认证绕过 |

### 3. 输入验证问题

| 问题 | 位置 | 风险 |
|------|------|------|
| 多处使用 `json.NewDecoder` 而非 `common.DecodeJson` | `controller/user.go` 多处 | 违反项目规范 Rule 1 |
| `UpdateSelf` 使用 `map[string]interface{}` 接收输入 | `controller/user.go:727` | 绕过结构体验证 |
| 密码占位符 `"$I_LOVE_U"` 绕过验证器 | `controller/user.go:652` | 回滚逻辑若被跳过可致意外行为 |

### 4. 硬编码凭据

| 位置 | 内容 | 风险 |
|------|------|------|
| `scripts/dev-api-sqlite.ps1:34` | `$env:SESSION_SECRET = 'local-dev-session-secret-change-in-production'` | 误用于生产可致 session 伪造 |
| `middleware/auth.go:148` | `c.Header("Auth-Version", "864b7076dbcd0a3c01b5520316720ebf")` | 版本指纹泄露 |
| `middleware/auth.go:381` | `"midjourney-proxy"` 特殊密钥 | 可能被滥用 |

### 5. 错误处理信息泄露

| 位置 | 泄露内容 |
|------|---------|
| `controller/user.go:144` | `err.Error()` 直接返回 |

| `controller/oidc_provider.go` | OAuth 错误详情 |
| `controller/setup.go:109,127` | 系统错误/创建管理员失败详情 |
| `controller/topup_waffo.go:344` | Webhook 签名和完整请求体记录到日志 |
| `controller/performance.go:83-140` | 服务器内存/GC/Goroutine/磁盘路径等（RootAuth 保护） |

### 6. SSRF 防护系统

项目实现了较完善的 SSRF 防护（`common/ssrf_protection.go`），包括：

- 私有 IP 检测（IPv4/IPv6 全覆盖）
- 域名白名单/黑名单
- IP 白名单/黑名单（CIDR 格式）
- 端口限制（默认仅允许 80, 443, 8080, 8443）
- DNS 解析后 IP 验证
- HTTP 重定向跟踪验证

**但以下位置绕过了 SSRF 防护**：

| 位置 | 说明 |
|------|------|
| Relay 通道出站请求 | Channel BaseURL 不经 SSRF 验证 |
| `controller/channel.go:998` FetchOllamaModels | 用户提供的 baseURL 无 SSRF 防护 |
| `relay/channel/ollama/relay-ollama.go:288` | 同上 |
| SSRF 防护可被管理员禁用 | `FetchSetting.EnableSSRFProtection` 可设为 false |

## 五、Cloudflare 保护下的后台端口暴露风险分析

### 攻击路径分析

```
攻击者目标：绕过 CF 直接访问后台端口
                    │
    ┌───────────────┼───────────────┐
    │               │               │
    ▼               ▼               ▼
  路径1           路径2           路径3
 发现真实IP     利用信息泄露     利用应用漏洞
    │               │               │
    ▼               ▼               ▼
 DNS历史记录    /api/status      SSRF漏洞
 SSL证书搜索    响应头版本号      Ollama端点
 子域名扫描     错误信息泄露      Relay通道
 邮件头泄露                      Webhook URL
```

### 路径1：发现真实 IP 的可能性

| 方法 | 风险 | 说明 |
|------|------|------|
| DNS 历史记录 | 🟠 高 | 如果域名曾直接指向服务器 IP，历史记录可能仍可查到 |
| SSL 证书透明度日志 | 🟡 中 | 如果服务器有独立 SSL 证书，可通过 crt.sh 查找 |
| 邮件头发泄露 | 🟡 中 | 如果系统发送邮件且邮件服务器与 Web 服务器同 IP |
| 子域名扫描 | 🟡 中 | 其他子域名可能未接入 CF |
| 响应头/错误信息 | 🟠 高 | `/api/status` 泄露 `server_address`，响应头泄露版本号 |

### 路径2：信息泄露辅助攻击

`/api/status` 端点泄露的 `server_address` 可能直接暴露服务器真实 IP 或内网地址。如果管理员配置了内网地址作为 `ServerAddress`，攻击者可以：

1. 获取内网拓扑信息
2. 结合 SSRF 漏洞访问内网服务

### 路径3：应用层漏洞绕过 CF

CF 的 WAF 主要防御已知攻击模式（SQL 注入、XSS 等），但**无法防御**：

- CORS 劫持（S1）— 请求从用户浏览器发出，完全合法
- 认证逻辑漏洞 — 如 Token IP 限制绕过
- 业务逻辑漏洞 — 如 Webhook 伪造
- SSRF — 请求从服务器发出，不经过 CF

### 一旦真实 IP 被发现的后果

| 后果 | 严重程度 | 说明 |
|------|---------|------|
| DDoS 攻击直接打到源站 | 🔴 严重 | CF 的 DDoS 防护完全失效 |
| 绕过 WAF 规则 | 🔴 严重 | 所有 WAF 规则失效 |
| IP 限流绕过 | 🟠 高 | 伪造 `X-Forwarded-For` 即可 |
| Token IP 白名单绕过 | 🟠 高 | `ClientIP()` 可被伪造 |
| pprof 端口暴露 | 🔴 严重 | 如果 `ENABLE_PPROF=true` |
| 暴力破解不受限 | 🟠 高 | CF 的 Rate Limiting 失效 |

## 六、安全加固建议

### 🔴 紧急修复（立即）

**1. 配置 Gin TrustedProxies，验证 Cloudflare IP**

```go
// 在 main.go 中，server.Run 之前添加
trustedCFIPs := []string{
    // Cloudflare IPv4
    "173.245.48.0/20", "103.21.244.0/22", "103.22.200.0/22",
    "103.31.4.0/22", "141.101.64.0/18", "108.162.192.0/18",
    "190.93.240.0/20", "188.114.96.0/20", "197.234.240.0/22",
    "198.41.128.0/17", "162.158.0.0/15", "104.16.0.0/13",
    "104.24.0.0/14", "172.64.0.0/13", "131.0.72.0/22",
    // Cloudflare IPv6 (如需要)
}
server.SetTrustedProxies(trustedCFIPs)
```

**2. 修复 CORS 配置**

```go
// 替换 AllowAllOrigins = true
config.AllowOrigins = []string{"https://your-domain.com"}
config.AllowCredentials = true
```

**3. 防止直接 IP 访问 — 添加 Host 头检查中间件**

```go
func HostCheck() gin.HandlerFunc {
    return func(c *gin.Context) {
        host := c.Request.Host
        if !strings.Contains(host, "your-domain.com") {
            c.AbortWithStatus(444) // Nginx 风格的无响应断开
            return
        }
        c.Next()
    }
}
```

**4. 禁用 pprof 或绑定到 127.0.0.1**

```go
// 修改 main.go
if os.Getenv("ENABLE_PPROF") == "true" {
    gopool.Go(func() {
        log.Println(http.ListenAndServe("127.0.0.1:8005", nil))
    })
}
```

### 🟠 高优先级修复

**5.** 减小 `/api/status` 信息泄露 — 移除 `server_address`、OAuth Client ID 等敏感字段，或拆分为公开/认证两个版本

**6.** 统一错误处理 — 所有 `err.Error()` 替换为通用错误消息，详细信息仅记录到日志

**7.** WebSocket CheckOrigin 验证 — 检查 Origin 头部是否匹配预期域名

**8.** Webhook 端点增加 IP 白名单 — 限制只接受来自支付服务商 IP 段的请求

**9.** 添加服务器绑定地址配置 — 支持 `LISTEN_ADDR` 环境变量，允许绑定到 `127.0.0.1`

### 🟡 中优先级修复

**10.** Turnstile 验证增加过期机制 — Session 中存储验证时间戳，设置 1 小时过期

**11.** Setup 端点增加一次性 Token 保护 — 首次部署时生成随机 Token，必须携带才能初始化

**12.** Relay 通道出站请求增加 SSRF 防护 — 验证 Channel BaseURL 不指向私有 IP

**13.** 修复 SMTP TLS 证书验证 — 添加 `SMTP_INSECURE_SKIP_VERIFY` 环境变量控制

**14.** 移除响应头中的版本号 — 删除 `X-New-Api-Version` 头部

**15.** 修复 `turnstile-check.go` 中 `json.NewDecoder` 为 `common.DecodeJson` — 遵守项目规范 Rule 1

**16.** `WssAuth` 空实现应添加实际认证逻辑或移除

### 🛡️ 基础设施层面

**17. 防火墙规则** — 在服务器层面配置 iptables/安全组，仅允许 Cloudflare IP 访问 3000 端口

```bash
# 示例：仅允许 Cloudflare IP 访问
iptables -A INPUT -p tcp --dport 3000 -s 173.245.48.0/20 -j ACCEPT
iptables -A INPUT -p tcp --dport 3000 -s 103.21.244.0/22 -j ACCEPT
# ... 添加所有 CF IP 段
iptables -A INPUT -p tcp --dport 3000 -j DROP
```

**18. 完全阻止 8005 端口的外部访问**

```bash
iptables -A INPUT -p tcp --dport 8005 -s 127.0.0.1 -j ACCEPT
iptables -A INPUT -p tcp --dport 8005 -j DROP
```

## 七、总结

| 风险等级 | 数量 | 核心问题 |
|---------|------|---------|
| 🔴 严重 | 3 | CORS 任意跨域、`/api/status` 信息泄露、无 CF IP 验证 |
| 🟠 高 | 5 | pprof 暴露、Webhook 无认证、限流可绕过、错误信息泄露、WebSocket 劫持 |
| 🟡 中 | 10 | Turnstile 无过期、Setup 无保护、SSRF 防护不完整等 |
| 🟢 低 | 7 | 迁移 SQL 拼接、测试硬编码密钥、版本指纹泄露等 |

**最关键的结论**：虽然域名部署在 Cloudflare 下，但项目本身**没有任何机制验证请求是否确实来自 CF 代理**。一旦攻击者通过任何途径发现了服务器的真实 IP（而 `/api/status` 泄露的 `server_address` 恰好可能提供这一信息），就可以完全绕过 CF 的所有防护，直接攻击源站。建议优先修复 S1-S3 三个严重风险，并在服务器防火墙层面限制仅 CF IP 可访问。
