先给结论：

  ## 结论

  就当前这份仓库代码来看，它现在不能直接作为“标准 OAuth2/OIDC Provider（身份提供方）”给你的另一个项目接入。

  它当前实现的是：

  - 本地账号系统
  - 外部 OAuth/OIDC 登录接入方（client / relying party）
  - 外部 OAuth Provider 配置管理器

  而不是：

  - 给别的项目签发 authorization code / access token / id token
  - 提供 /.well-known/openid-configuration
  - 提供 jwks
  - 提供 userinfo
  - 提供标准 authorize/token 端点

  所以，如果你问：

  > “这个项目能不能现在就作为标准 OAuth2/OIDC 提供方来用？”

  答案是：

  不能，至少不是“开个配置就行”的那种不能。它缺的是一整层协议角色，不是缺一个开关。

  ———

  ## 我为什么这么判断：基于当前仓库代码的事实

  这次我没有联网，因为你要的是代码查询，本地仓库已经足够回答。
  另外我没有启动服务，也没有跑测试，下面结论基于静态代码审查。

  ### 1) 路由层没有 Provider 应该有的标准端点

  看 router/api-router.go，目前看到的是：

  - /api/oauth/state
  - /api/oauth/:provider
  - /api/user/register
  - /api/user/login
  - /api/user/self
  - /api/custom-oauth-provider/*

  这些都说明它在做：

  - 登录/注册
  - 外部 OAuth 回调处理
  - 本地用户管理
  - 配置“外部”的 OAuth/OIDC 提供方

  但没有看到标准 OIDC Provider 典型端点，比如：

  - /.well-known/openid-configuration
  - /.well-known/jwks.json
  - /oauth/authorize
  - /oauth/token
  - /userinfo

  这已经是一个非常直接的信号：
  它不是 IdP。

  ———

  ### 2) controller/oauth.go 的流程明确是“去别处登录”

  看 controller/oauth.go 的 HandleOAuth 逻辑，大致是：

  1. 读取 provider
  2. 校验 state
  3. 调 provider.ExchangeToken(...)
  4. 调 provider.GetUserInfo(...)
  5. 本地 findOrCreateOAuthUser(...)
  6. setupLogin(...) 建立本地 session

  这说明它干的是：

  > “拿外部 provider 回来的 code，去外部 token endpoint 换 token，再取用户资料，然后映射成本地用户”

  这就是OAuth/OIDC 客户端逻辑，不是 Provider 逻辑。

  如果它是 Provider，流程应该相反：

  > “我接收别的项目带来的 authorize 请求，我给它发 code，我自己再签发 token / id_token / userinfo / jwks”

  当前代码完全不是这个方向。

  ———

  ### 3) oauth/oidc.go 证明它的 OIDC 是“客户端接别人”

  oauth/oidc.go 里很关键：

  - ExchangeToken(...) 使用的是配置里的 TokenEndpoint
  - GetUserInfo(...) 使用的是配置里的 UserInfoEndpoint

  也就是说，它是在调用别人的 OIDC 服务。

  这不是“自己作为 OIDC Provider”，而是“把自己当 OIDC Client”。

  ———

  ### 4) setting/system_setting/oidc.go 只是客户端配置，不是 Provider 配置

  这个配置结构里只有：

  - enabled
  - client_id
  - client_secret
  - well_known
  - authorization_endpoint
  - token_endpoint
  - user_info_endpoint

  你会注意到，这里没有 Provider 需要的东西，比如：

  - issuer
  - signing keys
  - jwks 配置
  - client registry
  - redirect URI registry
  - consent 配置
  - code/token 生命周期配置

  这说明它只是“如何接入一个外部 OIDC”的配置。

  ———

  ### 5) controller/misc.go 输出给前端的是“去外部登录”的信息

  GetStatus 暴露的字段包括：

  - oidc_enabled
  - oidc_client_id
  - oidc_authorization_endpoint
  - custom_oauth_providers

  这些是给前端登录按钮用的，让前端跳到外部登录页面。

  不是给别的项目读取的 OIDC Discovery 文档。

  ———

  ### 6) 用户模型是“绑定外部身份”，不是“管理外部 client”

  model/user.go 里有这些字段：

  - GitHubId
  - DiscordId
  - OidcId
  - WeChatId
  - TelegramId
  - LinuxDOId

  这说明它支持：

  - 一个本地用户绑定多个外部身份源

  但这和“我作为 IdP 管理别人注册到我这里的 OAuth Client”完全是两回事。

  ———

  ### 7) 当前的 access_token 也不是 OIDC token

  controller/user.go 里的 GenerateAccessToken 生成的是随机字符串；
  middleware/auth.go 里也能看出这是本系统内部鉴权用的 access token。

  它不是标准 OIDC 意义上的：

  - access token
  - refresh token
  - id token

  而且这个系统的鉴权还夹带了它自己的约定，例如 New-Api-User 头。
  这更说明：

  它的认证接口是“本项目私有协议”，不是通用 IdP 协议。

  # 如果你要让另一个项目“标准接入”这个项目，应该怎么做？

  我建议你把方案拆成两层来看：

  ———

  ## 方案 A：短期最实用

  ## 先不要强行做标准 OIDC，而是做“受控的自定义接入”

  如果两个项目都在你自己控制下，短期最稳的是：

  - 当前项目仍作为统一账号中心
  - 另一个项目不要直接复用当前所有 dashboard API
  - 而是新增一组明确为跨项目集成准备的 API

  例如：

  ### 当前项目新增的集成接口

  - POST /api/auth-hub/session/exchange
  - GET /api/auth-hub/me
  - GET /api/auth-hub/entitlements
  - POST /api/auth-hub/license/issue
  - POST /api/auth-hub/license/refresh
  - POST /api/auth-hub/license/verify

  ### 另一个项目做的事

  - 引导用户去当前项目登录
  - 登录完成后拿到一个一次性 code / session proof
  - 用这个 proof 去换本地 session
  - 本地建立 external_account_id -> local_user_id 的映射
  - workspace/project 仍然由另一个项目自己管理

  ### 这个方案的优点

  - 快
  - 侵入小
  - 不用一次性把 OIDC 全补齐
  - 非常适合你当前“两个项目都在自己手里”的情况

  ### 缺点

  - 不是标准协议
  - 以后第三方系统接入时复用性差

  ———

  ## 方案 B：中长期标准化

  ## 把当前项目补成一个真正的 OIDC Provider

  如果你明确希望：

  - 另一个项目按标准 OIDC 接入
  - 未来可能还有第 3 个、第 4 个项目接入
  - 桌面端、Web 端、移动端、后台都共享统一登录协议

  那就应该做 OIDC。

  而且我认为从你的系统演进方向看，这件事值得做，只是要分阶段做。

  # 如果要补成 OIDC Provider，最少要补哪些能力？

  下面是最小可行集合。

  ## 1) Discovery

  提供：

  - GET /.well-known/openid-configuration

  返回：

  - issuer
  - authorization_endpoint
  - token_endpoint
  - userinfo_endpoint
  - jwks_uri
  - scopes_supported
  - response_types_supported
  - grant_types_supported
  - code_challenge_methods_supported

  ———

  ## 2) Authorization Endpoint

  提供：

  - GET /oauth/authorize

  能力包括：

  - 校验 client_id
  - 校验 redirect_uri
  - 校验 scope
  - 校验 state
  - 校验 nonce
  - 支持 PKCE：code_challenge
  - 判断用户是否已登录
  - 未登录则跳登录
  - 已登录则展示 consent / 或跳过 consent
  - 签发 authorization code

  ———

  ## 3) Token Endpoint

  提供：

  - POST /oauth/token

  至少支持：

  - authorization_code
  - refresh_token

  返回：

  - access_token
  - token_type
  - expires_in
  - refresh_token
  - id_token

  注意：

  这里的 token 不能复用当前系统内部的 model.Token。
  因为语义完全不同。

  ———

  ## 4) UserInfo Endpoint

  提供：

  - GET /userinfo

  返回标准 claims，例如：

  - sub
  - email
  - email_verified
  - name
  - preferred_username
  - picture

  ———

  ## 5) JWKS

  提供：

  - GET /.well-known/jwks.json

  用于让另一个项目验证 id_token 签名。

  ———

  ## 6) Client 管理

  新增 OAuth Client 管理能力，至少包括：

  - client_id
  - client_secret
  - name
  - redirect_uris
  - allowed_scopes
  - grant_types
  - status

  这部分当前仓库没有现成模型，需要新增。

  ———

  ## 7) Consent

  新增用户授权记录：

  - 用户给哪个 client 授过哪些 scope
  - 是否允许跳过再次确认
  - 是否撤销过

  ———

  ## 8) Token 生命周期与安全

  至少补：

  - code 一次性消费
  - access token 过期
  - refresh token 轮换
  - revoke
  - introspection（可选，但推荐）
  - 签名密钥轮换
  - 审计日志
  - anti-replay
  - nonce / PKCE

  ———

  ### 1) User 主体模型

  - 不一定直接把数据库自增 id 暴露出去

  ———

  ### 2) 登录态 session

  controller/user.go、middleware/auth.go

  现有 session 可以继续用来支撑：

  - 登录页
  - 授权页
  - 2FA / Passkey 验证后的会话
  - consent 流程

  这个复用价值很高。

  ———

  ### 3) 现有外部 OAuth 登录聚合能力

  controller/oauth.go、oauth/*.go

  这部分可以继续保留，作为 Auth Hub 自己的“上游身份源接入”。

  也就是：

  - 当前项目未来作为 IdP
  - 但它自己仍然可以允许用户用 GitHub / Discord / OIDC 登录
  - 登录后统一映射到当前项目本地用户

  这个很合理。

  ———

  ### 4) 用户权益模型

  model/user_product_entitlement.go、controller/user_entitlement.go

  这个很值得保留。

  它天然适合作为：

  - 用户拥有哪些产品资格
  - 哪些资格启用/禁用
  - 哪些资格何时过期

  但我建议：

  - 不要把大量动态权益直接塞进 id_token
  - 更适合作为：
      - /userinfo 的少量稳定 claim
      - 或独立的 entitlements API

  因为权益变化频繁，而 token 会缓存，容易过时。

  ———

  ### 5) 注册码 / 订阅 / 运营后台能力

  这部分很适合继续留在当前项目里，作为“统一账号与权益后台”。

  ———

  ## 不建议直接复用的部分

  ### 1) model.Token

  它是网关业务 token，不是 OIDC token。

  你可以借鉴它的“存储和生命周期管理思路”，
  但不要直接拿它当 OAuth access token / refresh token / id token。

  ———

  ### 2) 当前 dashboard 私有鉴权约定

  比如 New-Api-User 这种约定，不适合作为跨项目标准身份协议的一部分。

  # 另一个项目如果按 OIDC 接入，实际流程应该是什么？

  ## Web 项目接入流程

  ### 第一步：在当前项目注册一个 OAuth Client

  为另一个项目生成：

  - client_id
  - client_secret
  - redirect_uri 列表
  - allowed scopes

  ———

  ### 第二步：另一个项目走 Authorization Code + PKCE

  浏览器跳转：

  - authorize

  用户在当前项目完成：

  - 登录
  - 2FA / Passkey
  - 同意授权

  回调到另一个项目的：

  - redirect_uri

  ———

  ### 第三步：另一个项目换 token

  服务端调用：

  - token endpoint

  得到：

  - access_token
  - id_token
  - refresh_token

  ———

  ### 第四步：另一个项目验证 id_token

  通过当前项目的：

  - jwks_uri

  验证签名和 claims。

  ———

  ### 第五步：另一个项目建立本地映射

  建立：

  - external_sub -> local_user_id

  注意：

  这里是用户映射，不是工作区映射。

  ———

  ### 第六步：另一个项目自己管理 workspace/project

  不要把 workspace 概念塞进 OIDC 身份层。

  建议：

  - sub 只代表用户
  - workspace 仍由另一个项目内部维护

  ———

  ### 第七步：按需拉取权益

  如果需要判断产品资格：

  - 可以调用 Auth Hub 的 entitlements API
  - 或在 userinfo 里放少量稳定 claim

  但不要把复杂动态权益全部压进 token。

  ———

  # 桌面混合模式怎么结合进来？

  你前面说的“第一次联网登录，之后可离线使用，定期复核”，其实非常适合和上面的架构组合。

  我建议是：

  ## OIDC 负责“人是谁”

  - 用户登录
  - 身份确认
  - 获得统一账号 sub

  ## Entitlement 负责“这个人有什么权利”

  - 有没有小说产品资格
  - 有没有 Pro / VIP / 订阅
  - 是否被禁用

  ## License 负责“这台设备能否离线继续运行”

  - machine_id
  - license_id
  - product_key
  - issued_at
  - expires_at
  - next_recheck_at
  - signature

  这样三层就分开了：

  - 用户身份：OIDC
  - 业务权益：Entitlement API
  - 设备离线许可：License API + 本地签名许可证

  这是我认为最稳的模型。
